# Phase H — 전진기지 UI와 정산

## 1. 개요

`process-B`의 B-7을 구현한다. A가 판정한 전진기지 Runtime 상태를 B의 충전, 보관함, 정산, 체크포인트 State와 UI에 연결한다.

## 2. 작업 목표

- 전력 공급/소비와 연결된 시설 및 비활성 원인을 표시한다.
- 충전, 플레이어 화물↔보관함 이동과 정산을 안전한 Service 경로로 처리한다.
- 전진기지 설치 완료를 체크포인트와 자동 저장 요청에 반영한다.
- 역할 경계를 지켜 A의 연결/거리/활성 판정을 재구현하지 않는다.

## 3. 구현 범위

- `Scripts/App/State/OutpostState` 또는 RunState 내 전진기지 상태
- `Scripts/App/Economy/OutpostService`, `Scripts/App/UI/Outpost/`
- `Prefabs/UI/OutpostPanel.prefab`
- `Tests/EditMode/App/Outpost/`, `Tests/PlayMode/Integration/Outpost/`

## 4. 권장 구현 방향

1. A가 제공하는 기지 ID, 활성 여부, 상호작용 가능 거리, 전력 공급/소비, 연결 시설과 원인 DTO를 확인한다.
2. OutpostPanel은 상호작용 가능 이벤트에서 열고, 거리 이탈·기지 파괴·Scene 종료 시 닫는다.
3. 충전은 A의 활성/전력 가능 판정 결과를 확인한 뒤 PlayerState의 현재 전력을 최대 전력 이하로 변경한다. 충전량과 소비량의 책임을 계약에 맞춘다.
4. 보관함은 플레이어 화물과 별도 ID별 수량 컬렉션으로 둔다. 이동 요청 전체를 사전 검증하고 출발지 차감과 도착지 증가를 원자적으로 수행한다.
5. 정산은 정산 대상(플레이어/보관함)을 명시하고, 실제 수량과 카탈로그 가격으로 합계를 계산한 뒤 광물 제거와 골드 지급을 한 트랜잭션으로 처리한다.
6. 이미 정산한 광물이 다시 판매되지 않게 정산 성공 뒤 원천 수량을 0으로 만들고 거래 이벤트를 한 번 발행한다.
7. 전진기지 설치 성공 이벤트에서 체크포인트 ID/필요 위치 DTO를 State에 기록하고 Phase K가 소비할 자동 저장 요청을 보낸다.
8. 첫 설치 시에만 간단한 튜토리얼을 표시하며, 표시 여부는 ProgressState에 저장 가능한 플래그로 둔다.

## 5. 보안 및 안정성 기준

- 연결 여부와 상호작용 거리는 A의 결과를 신뢰하고 B에서 물리 판정을 복제하지 않는다.
- 보관함 이동·정산 실패 시 양쪽 인벤토리와 골드가 모두 유지된다.
- Unity GameObject를 체크포인트나 저장 State에 넣지 않고 영구 ID와 DTO 값만 둔다.
- 중복 설치/정산 이벤트에 대한 멱등성 경계를 검증한다.

## 6. 완료 기준

- 기지 전력과 비활성 이유가 실제 Runtime 상태와 일치한다.
- 충전 후 전력이 최대치를 넘지 않고 즉시 표시된다.
- 보관함 수량이 플레이어 인벤토리와 별도로 정확히 관리된다.
- 정산이 한 번만 골드로 전환되고 설치 후 자동 저장이 요청된다.
