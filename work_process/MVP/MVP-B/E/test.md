# Phase E Agent Test

## 1. 정적 검증 항목

| ID | 테스트 항목 | 검증 방법 | 예상 결과 |
| :-- | :-- | :-- | :-- |
| E-S01 | 계약 역할 | `IResourceWallet` 구현 검토 | `CanAfford` 무변경, `TrySpend` 성공 시만 변경 |
| E-S02 | 가격 원천 | 판매 코드 확인 | 카탈로그 가격만 사용 |
| E-S03 | 원자성 | 거래 적용 경로 확인 | 부분 차감/부분 지급 경로 없음 |
| E-S04 | 중복 비용 | 합산 로직 테스트 확인 | 같은 ID 비용을 합산 후 검사 |
| E-S05 | UI 분리 | 버튼 핸들러 확인 | State 직접 변경 없음 |

## 2. 기능 테스트 항목

### E-F01: 일부 광물 판매

- **준비:** 구리와 철을 보유하고 가격을 고정한다.
- **실행:** 구리 일부만 판매한다.
- **예상 결과:** 구리만 감소하고 철은 유지되며 골드는 `수량×가격`만큼 증가한다.

### E-F02: 복수 비용 지불

- **준비:** 두 광물로 구성된 시설 비용과 충분한 재고를 둔다.
- **실행:** `CanAfford` 후 `TrySpend`를 호출한다.
- **예상 결과:** 첫 호출은 무변경이고 두 번째 호출만 모든 비용을 한 번 차감한다.

### E-F03: 자원 부족 원자성

- **준비:** 비용 중 한 광물만 부족하게 둔다.
- **실행:** `TrySpend`를 호출한다.
- **예상 결과:** false가 반환되고 모든 광물 수량과 골드가 이전과 같다.

### E-F04: 설치 실패

- **준비:** 비용은 충분하지만 A 배치 대역이 생성 실패를 반환하게 한다.
- **실행:** 건설 흐름을 끝까지 수행한다.
- **예상 결과:** `TrySpend`가 호출되지 않고 재고가 유지된다.

### E-F05: 중복 요청

- **준비:** 판매 버튼을 활성화하고 처리 횟수를 기록한다.
- **실행:** 같은 프레임에 버튼을 여러 번 누른다.
- **예상 결과:** 허용된 요청 한 건만 처리되며 지급/차감이 중복되지 않는다.

## 3. 테스트 절차

1. Edit Mode에서 판매, 비용 합산, 부족, 잘못된 값, 오버플로, 이벤트 횟수를 실행한다.
2. Play Mode에서 판매·제작 UI의 성공/실패 메시지와 중복 클릭을 확인한다.
3. A 배치 대역의 성공/실패 각각에서 자원 상태를 비교한다.
4. 성공 시 자동 저장 요청 이벤트가 한 번 발생하는지 기록한다.

## 4. 검증 결과 요약

- **모든 항목 통과 시:** Phase E 완료. 건설과 Surface Base는 같은 거래 API를 사용한다.
- **실패 항목 존재 시:** 검증 전 변경 여부와 중복 이벤트를 우선 확인하고 원자성이 입증될 때까지 완료 처리하지 않는다.
