# Phase E — 경제, 판매와 제작

## 1. 개요

`process-B`의 B-4를 구현한다. 광물 판매와 시설 제작 비용 검사를 Service 트랜잭션으로 처리하고, 설치 성공 전에는 자원을 차감하지 않는다.

## 2. 작업 목표

- 선택한 광물만 판매하고 정확한 골드를 지급한다.
- `IResourceWallet`로 시설 비용의 지불 가능 여부와 실제 차감을 제공한다.
- 판매·제작 성공/실패 결과를 UI와 자동 저장 요청에 전달한다.
- 중간 실패 시 인벤토리와 골드가 부분 변경되지 않게 한다.

## 3. 구현 범위

- `Scripts/App/Economy/`: `EconomyService`, `CraftingService`, 거래 결과/이벤트
- `Scripts/App/UI/Economy/`: 판매·제작 presenter
- Phase D Inventory와 Phase B Recipe/BuildingData 사용
- `Tests/EditMode/App/Economy/`, `Tests/PlayMode/Integration/Economy/`

## 4. 권장 구현 방향

1. 판매 요청은 광물 ID와 양수 수량을 받고, 카탈로그 가격·보유량·골드 오버플로를 모두 사전 검증한다.
2. 검증이 끝나면 인벤토리 차감과 골드 증가를 하나의 성공 경로에서 수행한다. 둘 중 하나라도 실패할 수 있다면 적용 전 스냅샷 또는 명시적 트랜잭션 API로 원자성을 보장한다.
3. `CanAfford`는 읽기 전용이고 State를 바꾸지 않는다. `TrySpend`는 모든 비용을 다시 검증한 뒤 전체를 한 번에 차감한다.
4. 같은 ID가 비용 목록에 여러 번 나오면 먼저 합산하고 오버플로와 총보유량을 검증한다.
5. A의 배치 시스템은 위치 검증과 Runtime Prefab 생성 성공 후에만 `TrySpend`를 호출한다. 성공 전 예약이 필요하면 Shared 합의 없이는 새 2단계 결제 계약을 만들지 않는다.
6. 판매/제작 UI는 요청 중복 클릭을 막고, 성공·자원 부족·잘못된 데이터·설치 실패를 구분해 표시한다.
7. 성공 이벤트에 거래 종류와 변경된 ID/수량을 넣고 Phase K의 자동 저장 요청 지점으로 연결할 수 있게 한다.
8. 실패 이벤트는 상태 이벤트를 발생시키지 않으며 디버그 로그와 사용자 메시지의 상세 수준을 분리한다.

## 5. 보안 및 안정성 기준

- 비용과 가격은 UI 입력이 아니라 검증된 카탈로그 값을 사용한다.
- 음수 비용, 빈 ID, 중복 비용과 정수 오버플로를 거부한다.
- `CanAfford` 호출만으로 자원을 예약하거나 차감하지 않는다.
- 설치 실패와 예외 후 원래 State가 보존되어야 한다.

## 6. 완료 기준

- 판매 수량만 감소하고 골드가 정확히 증가한다.
- 복수 자원 비용을 전부 보유할 때만 전체 차감된다.
- Runtime Prefab 생성 실패 시 자원이 유지된다.
- 거래 결과가 UI, State 이벤트와 저장 요청에 한 번 반영된다.
