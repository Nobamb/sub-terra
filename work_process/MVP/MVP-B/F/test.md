# Phase F Agent Test

## 1. 정적 검증 항목

| ID | 테스트 항목 | 검증 방법 | 예상 결과 |
| :-- | :-- | :-- | :-- |
| F-S01 | MVP 항목 완전성 | UpgradeData 목록 확인 | 7개 MVP 효과 정의 존재 |
| F-S02 | 정의/상태 분리 | 필드 검토 | 현재 레벨은 State에만 존재 |
| F-S03 | Provider 경계 | 참조 그래프 확인 | Gameplay가 App 구체 클래스를 참조하지 않음 |
| F-S04 | 구매 순서 | 코드/테스트 확인 | 모든 검증 뒤 결제와 레벨 상승 |
| F-S05 | 영구 ID | 카탈로그 검증 | 중복/빈 업그레이드 ID 없음 |

## 2. 기능 테스트 항목

### F-F01: 정상 구매와 효과

- **준비:** 레벨 0 업그레이드와 충분한 비용을 둔다.
- **실행:** 한 단계를 구매하고 Provider를 조회한다.
- **예상 결과:** 비용이 한 번 차감되고 레벨 1 효과가 반환된다.

### F-F02: 최대 레벨

- **준비:** 업그레이드를 최대 레벨로 설정한다.
- **실행:** 추가 구매를 요청한다.
- **예상 결과:** 실패 이유가 반환되고 비용·레벨·이벤트가 변하지 않는다.

### F-F03: 단계 데이터 누락

- **준비:** 다음 레벨 비용 또는 효과가 없는 테스트 데이터를 만든다.
- **실행:** 구매를 요청한다.
- **예상 결과:** 결제 전 실패하며 검증 도구가 누락 단계를 보고한다.

### F-F04: Gameplay 효과 반영

- **준비:** Provider를 사용하는 A 시스템 대역을 연결한다.
- **실행:** 업그레이드 전후 같은 입력으로 계산한다.
- **예상 결과:** 정의된 배율/최대치만큼 결과가 달라진다.

### F-F05: 심층 잠금

- **준비:** 잠금 조건 직전/충족 진행도를 각각 만든다.
- **실행:** 접근 가능 여부와 이유를 조회한다.
- **예상 결과:** 경계에서만 잠금이 해제되고 UI용 이유가 일치한다.

## 3. 테스트 절차

1. Edit Mode에서 각 효과의 레벨 0, 중간, 최대, 범위 밖 계산을 실행한다.
2. 구매 성공/자원 부족/최대 레벨/데이터 누락의 상태 원자성을 비교한다.
3. Play Mode에서 구매 UI, HUD 최대치와 A Provider 대역을 함께 확인한다.
4. 저장 연결 전에는 UpgradeState 스냅샷 직렬화 가능 여부를 확인하고, Phase K 후 재실행 유지 테스트를 추가한다.

## 4. 검증 결과 요약

- **모든 항목 통과 시:** Phase F 완료. Phase K와 L에서 저장·Surface Base 흐름을 연결한다.
- **실패 항목 존재 시:** 결제, 레벨 State, 효과 계산, Provider 소비자 중 실패 경계를 분리해 재검증한다.
