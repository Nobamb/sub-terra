# Phase D — 인벤토리와 화물 시스템

## 1. 개요

`process-B`의 B-3을 구현한다. A의 채굴 시스템이 전달한 광물 ID와 수량을 인벤토리에 반영하고, 데이터 카탈로그를 기준으로 총중량과 미정산 가치를 일관되게 계산한다.

## 2. 작업 목표

- `IMiningRewardReceiver`의 B 측 구현체를 제공한다.
- 광물별 수량, 현재/최대 화물 중량과 미정산 가치를 관리한다.
- 최대 적재량, 잘못된 ID·수량과 중복 지급 경계를 명시한다.
- 인벤토리와 HUD가 한 번의 상태 변경 결과를 즉시 표시한다.

## 3. 구현 범위

- `Scripts/App/Inventory/`: `InventoryState`, `InventoryService`, 계산 결과와 이벤트
- `Prefabs/UI/InventoryPanel.prefab`
- `Tests/EditMode/App/Inventory/`, `Tests/PlayMode/Integration/Inventory/`
- Shared의 `IMiningRewardReceiver`는 사용만 하며 변경하지 않는다.

## 4. 권장 구현 방향

1. 광물별 수량은 영구 ID와 0 이상의 정수로 저장한다. 없는 항목과 수량 0 항목을 저장/표시할 정책을 하나로 정한다.
2. 총중량과 미정산 가치는 `수량 × MineralData 단위값`의 합으로 한 곳에서 계산한다. UI나 Save DTO가 별도 계산하지 않는다.
3. `AddMineral`은 ID 존재, 양수 수량, 정수 오버플로, 잔여 적재량을 순서대로 검증한다.
4. 초과 지급 정책은 MVP 전체에서 하나로 고정한다. 권장값은 “들어갈 수 있는 완전한 단위만 수락하고 수락량/거절량을 결과로 반환”이며, Shared가 `void`라면 상세 결과는 B 내부 이벤트와 로그로 남긴다.
5. 지급 ID가 필요하다는 Shared 이벤트가 있으면 처리 완료 ID를 Run 단위로 기록해 중복 지급을 막는다. 계약에 지급 ID가 없다면 추측으로 중복 제거하지 말고 A에게 계약 보완을 요청한다.
6. 보관함 이동, 정산 차감 등 감소 연산은 공통 원자적 메서드를 사용한다. 일부 항목만 먼저 바꾼 뒤 실패하지 않게 사전 검증한다.
7. 성공한 변경 뒤 정확히 한 번 `InventoryChanged`를 발행하고 Phase C HUD와 InventoryPanel이 현재 스냅샷을 다시 그린다.
8. 테스트에서는 실제 ScriptableObject 대신 최소 카탈로그를 만들어 가격·중량 경계와 오버플로를 검증한다.

## 5. 보안 및 안정성 기준

- 알 수 없는 ID와 음수/0 지급은 상태를 바꾸지 않는다.
- 외부에 mutable Dictionary를 노출하지 않는다.
- 부동소수점 중량은 비교 오차 정책을 정하고, 가격/골드는 정수형 범위와 오버플로를 검증한다.
- 실패 로그에 전체 세이브나 사용자 경로를 포함하지 않는다.

## 6. 완료 기준

- 정상 보상은 광물별 수량, 중량, 가치에 정확히 한 번 반영된다.
- 적재량을 넘지 않고 수락/거절 결과를 확인할 수 있다.
- 알 수 없는 ID와 잘못된 수량에도 게임이 중단되지 않는다.
- 인벤토리 UI와 기본 HUD가 같은 State를 즉시 표시한다.
